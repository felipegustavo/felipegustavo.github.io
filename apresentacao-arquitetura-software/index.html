Desing de Software

Refere-se ao processo de transformar as demandas do usuário (requisitos) em uma especificação de software que vai guiar desenvolvedores na implementação da solução. Tem enfoque na solução do problema e planejamento do software em um nível mais baixo.

Arquitetura de Software

Refere-se a estrutura geral de um sistema sem se procupar com detalhes de implementação. Tem foco na divisão do sistema em componentes, na organização desses componentes e como eles se comunicam.

Design e Arquitetura na Prática

  • São a mesma coisa na prática
  • Os detalhes servem de base para as decisões de alto nível e vice-versa
  • Baixo nível e alto nível fazem parte do mesmo todo
  • Não há uma linha clara que separa eles, há apenas uma linha constante corta os dois níveis

Qual o Objetivo?

O objetivo da arquitetura é minimizar os recursos humanos necessários construir e manter um software.

Como medir a qualidade?

É a medida do esforço necessário para satisfazer as demandas do cliente.

Princípios de Design

Progrmação Orientada a Objetos

Os 4 Pilares

Abstração

Esconda detalhes de implementação dentro de alguma coisa.

Encapsulamento

Esconda alguma coisas no seu código numa capsula.

Herança

Reduça a quantidade de código reaproveitando comportamentos e propriedades.

Polimorfismo

Torne o seu código mutável de uma forma mais elegante e sustentável.

Padrões de Projeto

Os 6 padrões de todo arquiteto de software deve saber.

Singleton

Mantehna uma única referência do seu objeto compartilhada.

Facade

Simplifique o acesso a uma estrutura mais complexa de código.

Factory

Reduza a quantidade de dependências e simplifique a criação de objetos deixando que outros o façam.

Strategy

Torne o seu código mutável fornecendo uma família de algoritmos intercambiaveis.

Observer

Mantenha o seu estado consistente de forma simples e centralizada.

Adapter

Tenha mecanismos que permitam que componentes possam se comunicar sem se preocupar com os detalhes.

Princípios SOLID

Torne o seu design mais entendível, flexível e sustentável

S: Princípio da Responbilidade Única

Um módulo deve ter uma, e apenas uma razão para mudar.

Um módulo deve ser responsável por um, e apenas um, usuário ou stakeholder.

Um módulo deve ser responsável por um, e apenas um ator.

L: Principio da Substituição de Liskov

Se para cada objeto o1 do tipo S existe um objeto o2 do tipo T tal que para todos os programas P definidos em termos de T, o comportamento de P não é modificado quando substituimos o1 por o2 então S é um subtipo de T.

Problema

Solução

I: Principio da Segregação de Interface

Evite depender de coisas que você não precisa.

D: Principio da Inversão de Dependência

Código que implementa políticas de alto nível não deve depender de código que implementa detalhes de baixo nível. Em vez disso, detalhes devem depender de políticas.

  • Não referencie classes concretas voláteis
  • Não derive de classes concretas voláteis
  • Evite override em funções concretas
  • Nunca menciona o nome de qualquer coisa concreta e volátil
Solução
Martin, Robert. Arquitetura Limpa: O Guia do Artesão para Estrutura e Design de Software; Traduzido por Samantha Batista. Rio de Janeiro: Alta Books, 2018.

Componentes

A nova onda do momento é montar o sistema como se fosse um lego.

Componentes

Um componente refere-se a menor parte implatável de um sistema.

Componentes

Mantenha o seu código pequeno, além de promover o reuso e paralelizar o processo de desenvolvimento.

O que faz um bom componente?

  • Ser reusável
  • Não o forçe a depender de coisas que não precisam
  • Deve ter apenas uma responsabilidade
  • Evite dependências ciclicas

Como fazer certo?

Caractéristicas de uma boa Arquitetura

  • Deve estabelecer fronteiras
  • Deve estabelecer políticas
  • Deve estabelecer níveis
  • Deve ser independente de frameworks, banco de dados, UI e outros agente externos
  • Deve ser testável
  • Deve ser gritante

Dívida Técnica

Refere-se a atividades que você deveria ter feito durante o projeto mas não as fez por algum motivo ou fez algo de uma forma não ideal. A dívida acumulada irá reduzir a produtividade do time e gerar dor de cabeça para os stakeholders.
